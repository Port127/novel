# daily-write（日更写作）

> **用途**：基于细纲逐章写作正文。通过质量门禁流水线确保输出质量。
> **前置条件**：
> - `settings/chapters_index.yaml` 存在（细纲已设计）
> **输出文件**：
> - `content/chapter_XXX.md`

---

## 1. 架构与行为规范 (System Rules)

> **[系统强制]** 本板块定义 Agent 必须遵守的操作流转规则，绝对不可违背或删减。
1. **防暴走与启发式交互 (UX)**：在非批量模式下，每个 Phase 结束前必须停下等待用户。提问时必须提供具体方案。在批量模式下，必须在 Phase 1 明确章节范围后才可连续运行。
2. **多智能体编排 (Orchestration)**：在正文生成（Phase 3）阶段，主 Agent 必须使用 `invoke_subagent` 唤醒 `narrative-writer` 来负责具体的文本生成，主 Agent 仅负责流程统筹与最终落盘。
3. **上下文闭环 (Context Loop)**：强制使用 `/nm` 获取素材，并将其 100% 映射到写手的载荷中。
4. **实时进度保存 (State Persistence)**：进入任何一个新的 Phase，必须立即更新根目录下的 `_progress.md` 文件。

---

## 2. 创作与业务准则 (Domain Rules)

> **[业务核心]** 本板块定义该 Skill 独有的领域知识、创作契约和设定标准。重构时必须全量保留原有的业务逻辑，填入下方：

1. **断点恢复**：通过 `_progress.md` 记录写作进度，崩溃后可从断点续跑。
2. **质量门禁**：每章必须通过 JS 脚本检测 + LLM 评估双重验证。
3. **上下文追踪**：通过状态追踪文件保持章节间连贯性。
4. **反AI味**：生成的正文必须通过 AI 味检测，去除写作指纹。
5. **钩子密度**：每章末尾必须有悬念钩子，保持读者粘性。
6. **深度限知视角**：全程锁死主视角角色的此刻感知，读者与她同步获知，不提前剧透、不补全背景。
7. **情绪烈度**：网文要强噱头、强爽、强情绪。爽点要狠要具体，敢写极端反应，绝不点到为止（题材以克制为爽感的除外）。
8. **字数硬约束**：字数达标是硬性要求，未达标的章节视为未完成。写完第一件事就是验证字数。

---

## Phase 定义 (Phase State Machine)

> **【架构强制要求】**：
> 1. 生成任何结构化数据前，**必须使用 `view_file` 强制读取对应的 `data/schemas/*.schema.yaml`**。
> 2. 严禁按己意图捏造不在 Schema 中的顶级字段。
> 3. Reference 文件应当按需在具体的 Phase 中加载，不要在 Phase 1 一次性全读完。

### Phase 1：准备与上下文对齐（批量日更任务加载与前置检查）

**入口条件**：chapters_index.yaml 存在
**目标**：确定本次批量日更的目标章节，并加载历史上下文

**步骤**：
1. **进度更新**：更新 `_progress.md` 的 `current_phase` 为 1。
2. **上下文读取**：读取 `settings/notes.yaml` 或专属的 `settings/context.yaml`，加载过去积累的**三层上下文结构**（近5章详记、十章概要、卷级总览）。`settings/notes.yaml` 必须保持 `data/schemas/notes.schema.yaml` 的结构。如果文件不存在（第一次日更），从第 1 章开始创建。
3. **确认目标章节**：结合上下文最后记录的完成进度与 `settings/chapters_index.yaml`，向用户确认本次批量日更的章节范围（默认一次写 2-3 章）。
4. **断点检查**：检查 `_progress.md`，如有崩溃留下的未完成单章，提示先续跑该章。
5. **细纲与加载**：从 `settings/chapters_index.yaml` (章元数据) 与 `settings/arcs.yaml` (情节点) 中提取选定章节的细纲。
   - `settings/chapters_index.yaml` 是目标章节定位和跨 Skill 调度的主入口。若存在三位补零的 `settings/chapter_outlines/chapter_XXX.md`（例如 `chapter_001.md`），必须读取它作为单章详细蓝图；若不存在，则只能基于章节总索引、`settings/arcs.yaml`、`settings/outline.yaml`、前文正文和 `settings/notes.yaml` 补建临时细纲，不得跳过细纲直接盲写。
   - **【强制机制】细纲缺失补建**：若目标细纲不存在，绝不可跳过或盲写。必须强制读取 `data/schemas/outline.schema.yaml`、调用 `settings/outline.yaml` (全书大纲)、最新章节正文与 `settings/notes.yaml`，推演出本章的核心事件、出场角色与目标情绪，生成临时细纲后才可进入 Phase 2。
   - 如果存在 `paywall_report.yaml`，必须按 `data/schemas/paywall_report.schema.yaml` 读取 `paywall_chapter`、`final_cut.cliffhanger` 和 `final_cut.payoff_promise`，用于判断当前章节是否需要执行卡点前后承诺。

6. **停顿确认**：等待用户确认操作范围后进入下一阶段。

**出口条件**：操作范围已与用户确认，本次日更的目标范围和基础上下文已明确
**加载 References**：无

### Phase 2：核心任务拆解（强制文风召回与上下文组装）

**入口条件**：进入具体单章的写作前
**目标**：结合细纲与素材库 `nm` 组装精准的对标参数

**步骤**：
1. **进度更新**：更新 `_progress.md` 的 `current_phase` 为 2。
2. **状态重核**：读取上一章末尾 300 字，并在 `settings/notes.yaml` 下的 tracking 节点筛选本章涉及角色的当前状态和待回收伏笔。
3. **【核心供血】强制使用 `/nm` 召回素材（绝对不可跳过）**：
   - **获取原文锚点**：执行 `/nm search chapter --query "本章基调与高潮类型"` 检索并提取 300-500 字的优质原文片段，**严格赋值给变量 `[anchor_excerpts]`**。这是防止句式 AI 味的最强防御，没有对标原文片段绝不允许进入写作。
   - **获取套路骨架**：执行 `/nm search detail --query "本章核心事件与目标情绪"` 检索同品类小说，选定最贴切的拆解，**赋值给变量 `[selected_emotion_module]` 和 `[rhythm_reference]`**。
   - 若素材库未查到相关结果，必须提醒用户补充，或回退到全书设定的基础文风锚点。绝不允许“空载”参数进入写手阶段。
4. 读取 `references/state-tracking.md`，将上述内容与参数组装为单章的《写前备忘录》。

**出口条件**：单章的完整上下文与对标锚点准备就绪
**加载 References**：`state-tracking.md`

### Phase 3：写作执行

**入口条件**：上下文加载完成
**目标**：按细纲生成正文，字数达标

**步骤**：
0. **进度更新**：更新 `_progress.md` 的 `current_phase` 为 3。

#### 3.1 写前意图确认

1. **意图落定**：基于 Phase 2 召回的素材与本章细纲，用一句话严格写下本章的写作意图（必须包含：起因、情绪目标、核心逻辑线、拟采用的对标技法）。例如：“快节奏打脸——起因是账单暴露，逻辑线=发现→逼问→反证→公开代价；复现 nm 召回的‘信息差反杀’，标点和句长照抄原文锚点的短句节奏。”
2. **多线与配角安排核查**：确认本章是否分配了配角的反应镜头（不能只写主角的动作）。如果有装逼/打脸段落，必须明确哪些配角承担“集体震惊或各异”的反应位。

#### 3.2 场景写作

5. **场景写法 — 三维度揉进**（参考 `references/writing-craft.md` 第 8 节）
   - 深度限知视角：锁死主视角角色的此刻感知，只写她看到/听到/闻到/感到/脑中闪过的
   - 发生 + 感知 + 反应织在同一段，**不按维度分段**
   - 画面分段：按新动作/新物件/新信息断段，不按维度断段
   - 子事件按疏密分配：详写 ≥100-150 字，过场 1-2 句带过
   - **主观偏差代替客观叙述**：场景被角色情绪染色，她恨谁谁就可憎，她慌光线就晃
   - 镜头不拉远、不俯瞰、不切他人内心；读者与她同步获知
6. **身体细节替代情绪词**（参考 `references/writing-craft.md` 第 1 节）
   - 用具体身体反应替代"悲伤""愤怒"等抽象词
   - 情绪词默认外化，只保留紧跟身体反应的体感词
   - 同一身体部位全文 ≤5 次
7. **贯穿道具运用**（参考 `references/writing-craft.md` 第 2 节）
   - 检查本章是否涉及贯穿道具的三次出现编排
   - 每次出现意义需翻转递进（第 1 次建立→第 2 次强化→第 3 次意义翻转/回扣暴击）
   - 物件的初始情感含义设计参考 emotional-methods.md

#### 3.3 对话写作

8. **对话差异化**（参考 `references/dialogue-mastery.md`）
   - 潜台词与信息控制：角色真正想要的不浅显写在台词里
   - 权力模式：谁在掌控节奏（压制/反转/心死），用对话长度标记权力位
     - 权力方：掌控话题、打断、下结论（短句）
     - 弱势方：解释、求情、沉默（长句）
     - 反转：弱势方突然亮出底牌
   - 逐句情绪承接：禁止问答式/信息倾倒式对话
   - 语言风格档案 7 维度差异化：遮住名字能分出谁在说话
9. **对话中的信息展示**（参考 `references/dialogue-mastery.md`「信息展示与世界观引出」）
   - 通过冲突/争论自然引出设定，禁止角色当"科普嘴"
   - 信息不对称制造张力：A 知道 B 不知道的事
   - 能用对话 + 动作带出的信息不用叙述旁白

#### 3.4 情绪与节奏

10. **情绪弧线执行**（参考 `references/emotional-methods.md`）
    - 情绪烈度：爽点要狠要具体，敢写极端反应（对方失态、围观哗然），绝不点到为止
    - 拉扯节奏：有回落再升，不能一直升
    - 情弦理论：锁定目标读者的核心情感弦，每节至少拨一次
    - 数字/金额作为叙事工具（参考 `references/writing-craft.md` 第 4 节）：用具体数字增强冲击力
11. **节奏控制**（参考 `references/writing-craft.md` 第 5/7 节）
    - 一动一静节奏：动作段后接静止感知段
    - 小节密度诊断：爽点 beat 写密、过场 beat 写疏
    - 高潮压短句、沉淀放长句，忌通篇同长度
    - 输出前密度重排：扫描每段，按新动作/新物件/新信息/新对话拆开；连续碎段像提纲时合并

#### 3.5 格式与收尾

12. **格式遵守**（参考 `references/format-and-structure.md`）
    - 段落间无空行（`\n` 不是 `\n\n`）
    - 不用省略号/破折号，改用句号、逗号、动作 beat、短句或换行
    - 标点服务语气，避免通篇句号化
    - 主语节奏：段首点名建立主语、段中代词/省略、关键转折再点名
    - 对话独立成行，遵守平台对话格式

#### 3.6 字数验证（写完第一件事）

13. **统计字数**：`wc -m < 文件路径`，禁止 `wc -c`（字节数非字数）和模型估算
14. **字数不足**（< 章目标 90%）→ 绝不允许挤牙膏式加废话。必须定位欠账密点，回到细纲扩写情节点后一次性重写。
15. **字数超标**（> 章目标 ×1.1）→ 压过场、合并疏点，不得灌水
16. **正文元信息清洗**：检查正文是否漏入工程词（如 `第X章|上一章|本章|前文|后文|伏笔|细纲|读者`），命中强制改写为内世界感知（如“比上一章更疼”改为“比刚才开火更疼”）。
17. **对话自检**：正文含对话时逐句检查，命中即改：
    - 机械对话（问答式/句间无情绪承接）
    - 角色当"科普嘴"（整段讲设定/原理，Gate G）
    - 说话不分场合（高压 beat 的玩笑出戏）
18. **文风漂移检查**：续写时核对句长分布是否偏离 `[anchor_excerpts]` 目标，命中即拉回重写

**出口条件**：正文已生成，字数达标，元信息已清洗，对话与文风自检通过
**加载 References**：`writing-craft.md`、`dialogue-mastery.md`、`emotional-methods.md`、`style-craft.md`、`format-and-structure.md`、`hooks-paragraph.md`


#### Agent 调用编排 (Orchestration)：narrative-writer（强制约束型增强）

如果项目已部署 narrative-writer agent（检查 `.agents/agents/narrative-writer.md` 是否存在），可读取文件并必须拼接以下**带有强硬约束和详尽参数的 Payload** spawn Agent 执行写作：

- 项目根目录：{当前项目绝对路径}
- 任务描述：写正文，章节第 {N} 章，目标字数 {X}
- 核心意图：{Phase 3.1 确认的一句话写作意图，含情绪、节奏与对标套路}
- 上下文与状态：{本章涉及的角色最新状态快照、待回收伏笔、上一章核心事件摘要}
- **情节点预算契约**：{摘自细纲：每个情节点的详略分配与字数预算。密点（爽点/高潮）必须展开详写 >250字，疏点（过场）一笔带过。严格遵循。}
- **情绪模块与套路骨架 (selected_emotion_module)**：{Phase 2 召回的具体套路，必须复现其情绪逻辑而非抄袭剧情}
- **文风强约束**：{强调禁用省略号/双破折号/段间空行，强调三维度（发生+感知+反应）揉进同一段，明确按语气标点谱系执行，禁用先否定再肯定的AI句式}
- **【核心】原文锚点片段 (anchor_excerpts)**：{完整粘贴从 nm 召回的 300-500 字对标原文片段。硬性要求 Agent 必须模仿其句长分布、停顿节奏与信息释放速度，作为最高优先级的风格参考基准}

如 agent 不可用，由主线程直接扮演，但需在当前对话上下文中显式加载上述约束并确认。

### Phase 4：落盘验证一 (确定性收尾与退化防护)

**入口条件**：初稿生成
**目标**：生成最终文件并通过自动化门禁检查，运行 JS 脚本进行确定性检测 + 风格/钩子检查

**步骤**：
0. **进度更新**：更新 `_progress.md` 的 `current_phase` 为 4。
1. **AI 模式拦截**：运行 `node .agents/skills/_shared/scripts/check-ai-patterns.js --check content/chapter_XXX.md`。
   - 必须处理四类问题：① 破折号（不要一律改句号：打断改动作 beat、长音改动作、说明改逗号）；② 连续短句号（合并长句）；③ 先否定再肯定（删否定直奔肯定）；④ 段落过长。
   - 必须复扫正文直到 blocking 项归零。
2. **标点兜底**：运行 `node .agents/skills/daily-write/scripts/normalize-punctuation.js content/chapter_XXX.md`，自动清除残留的错误标点。
3. **退化防护（必须执行）**：运行 `node .agents/skills/_shared/scripts/check-degeneration.js --check content/chapter_XXX.md`。
   - 若命中 **硬信号 (blocking)**（复读、截断、漏入“作为AI”或“细纲/情节点”等底层工程词）：**带着报错证据回炉重写本章**（最多2次）。若仍失败则停止并报用户。
   - 若命中 **软信号 (advisory)**（“本章/下一章/伏笔/读者”）：需结合语境判断，若角色是在故事内真实使用则保留，否则修改。
4. 重新运行以上脚本直到 blocking 项全部归零。
5. **风格检查**（参考 `references/style-craft.md`）：
   - **打斗场景**：镜头式写作、白描、节奏骤短；一击必杀的爽感需铺垫反差
   - **装逼场景**：反差铺垫→一击必杀、围观者反应层级（从轻视到震惊）
   - **日常场景**：信息密度、潜台词、生活化比喻；禁止空洞无事的流水账
   - **文笔三要素**：画面感/节奏感/分寸感，逐段检查
   - **毒点检查**：是否有劝退读者的毒点（参考 style-craft.md「毒点解析与规避」）
7. **段落级钩子检查**（参考 `references/hooks-paragraph.md`）：
   - 每段是否有微钩子（11 种段落级钩子类型：悬念/好奇/信息差/反常/中断/承诺/威胁/诱惑/秘密/矛盾/未解）
   - 对话情绪是否递增（五级递增模型：平静→微澜→波动→激烈→爆发）
   - 围观者反应是否有质量层级（不是所有人都一样反应）
   - **不公平伤害钩子**：角色遭受不公平对待时，读者愤怒值是否被有效激发
   - **钩子禁忌检查**：是否有断章狗式的假悬念、是否有过度使用导致麻木的钩子

**出口条件**：所有脚本检测通过，风格/钩子检查通过
**加载 References**：`style-craft.md`、`hooks-paragraph.md`

### Phase 5：落盘验证二 (LLM 评估)

**入口条件**：脚本检测通过
**目标**：进行语义层面的质量评估

**步骤**：
0. **进度更新**：更新 `_progress.md` 的 `current_phase` 为 5。
1. 读取 `references/anti-ai-writing.md`，按 5 层标准评估：
   - 词汇层：是否有 AI 常用词（命运齿轮/如潮水般/仿佛春风等）
   - 句法层：是否有 AI 句式（连续排比/刻意对称/空洞抒情）
   - 段落层：是否有 AI 段落结构（总分总/过度对称/先否定再肯定）
   - 叙事层：是否有 AI 叙事模式（上帝视角/叙述者解释/剧透预告）
   - 情感层：是否有 AI 情感表达（抽象情绪词/万能比喻/升华式收尾）
2. 读取 `references/hooks-guide.md`，评估钩子质量：
   - 章首钩子：300 字内是否有第一个冲突/悬念
   - 章尾钩子：悬念强度（≥60 通过）
   - 冲突密度（≥60 通过）
   - 期待感循环：建立→维持→打破→重建 是否完整
3. **对话质量评估**（参考 `references/dialogue-mastery.md`「质量检查」）：
   - 是否有潜台词与议程碰撞
   - 权力模式是否清晰（谁在掌控）
   - 角色语言差异化是否达标（遮住名字能分出谁）
   - 信息展示是否通过对话自然引出
4. **格式合规检查**（参考 `references/format-and-structure.md`「5 条绝对禁止」+「快速检查」）：
   - 段落间是否无空行（`\n` 非 `\n\n`）
   - 标点是否服务语气（不堆砌、不通篇句号化）
   - 对话格式是否合规（独立成行、平台覆盖）
   - 是否无省略号/破折号
   - 主语节奏是否正确（段首点名/段中省略/转折点名）
5. 读取 `../_shared/references/quality-checklist.md`，逐项检查
6. 综合分 < 60 → 回到 Phase 3 重写
7. 读取 `references/banned-words.md`，检查禁词

**出口条件**：LLM 评估通过
**加载 References**：`anti-ai-writing.md`、`hooks-guide.md`、`quality-checklist.md`、`banned-words.md`

#### Agent 调用：consistency-checker（可选增强）

可读取该文件内容，拼接以下参数后 spawn Agent 做事实一致性扫描：

- 项目根目录：{当前项目绝对路径}
- 检查范围：content/chapter_XXX.md

如 agent 不可用，跳过此步。

### Phase 6：落盘验证三 (定稿与上下文折叠更新)

**入口条件**：所有检查通过
**目标**：输出最终文件、输出质量报告，并**执行三层上下文维护**。

**步骤**：
0. **进度更新**：更新 `_progress.md` 的 `current_phase` 为 6。
1. **最终通读**：快速通读全章，检查叙事流畅度、角色行为合理性、伏笔、章首章尾钩子是否到位。
2. 写入 `content/chapter_XXX.md` 并更新 `_progress.md` 标记单章阶段完成。
3. **【核心】更新状态追踪与三层上下文（长篇续写生命线）**：
   - 必须统一维护 `settings/notes.yaml`，并保持 `data/schemas/notes.schema.yaml` 的结构。不创建或修改任何零散 Markdown 文件。
   - **近5章详记**：追加本章核心事件、角色实质性变化与新增伏笔到 `tracking.recent_chapters`。若已有记录超过 5 章，必须**强制将最老的一章移出本层级**。
   - **十章概要**：当本章是 10 的整数倍时（如 10, 20, 30章），由主线程提炼过去 10 章的单句概要，写入 `tracking.ten_chapter_summaries`。
   - **卷级总览**：若本章是卷末收尾章，生成本卷的一句话总结，写入 `tracking.volume_overview`。
   - **角色与伏笔**：角色变化写入 `tracking.character_states`，伏笔状态写入 `tracking.foreshadowing`。
4. **批量循环**：展示本章质量报告。如本次批量日更任务尚未结束（例如计划写 3 章目前只写了 1 章），无需等待用户确认，直接流转回 Phase 2 进入下一章写作；否则结束流程。

**质量报告模板**：
```
## 第 N 章质量报告

```

**出口条件**：章节完成，质量报告生成

### 章节类型写作侧重

不同定位的章节，写作重心不同：

| 章节定位 | 写作侧重 | 情绪烈度 | 字数建议 | 特殊要求 |
|---------|---------|---------|---------|---------|
| **高压章** | 冲突密集、节奏快、短句为主 | 高 | 按目标 | 必须有明确的情绪峰值 |
| **推进章** | 信息推进、关系变化、伏笔埋设 | 中 | 按目标 | 不能只推剧情不顾情绪 |
| **修炼/试错章** | 能力提升、挫折、成长铺垫 | 中低 | 可略短 | 必须有至少一个小爽点或伏笔 |
| **关系章** | 互动、潜台词、情感升温 | 中 | 按目标 | 对话差异化必须到位 |
| **低压/生活章** | 日常、信息整理、过渡 | 低 | 可略短 | 不能灌水，每段至少推一项 |
| **高潮章** | 全面爆发、多线汇聚、节奏极快 | 极高 | 可超标 | 蓄能→假胜→崩解→真胜 |

**通用要求**：
- 低压章/过场章合计占比 ≤ 15%
- 相邻章不能情绪趋同（高压→低压→推进→高潮）
- 每章都要给读者一个往下看的理由

### 续写/改写

写作过程中可执行：
- **续写**：在已有正文基础上继续
  - 必须读取前章末 300 字 + `settings/notes.yaml` 的状态追踪节点
  - 续写衔接剧情，不衔接上一章的句式节奏
  - 文风漂移检查：核对句长分布是否偏离目标
- **改写**：修改已有正文（润色/扩充/精简/重写）
  - 扩充：补足细纲中遗漏的情节点，不得灌水
  - 精简：压缩过场/疏点，保留核心事件和情绪 beat
  - 重写：当章节质量无法通过门禁时，回到 Phase 3 重新执行
- **重新过审**：修改后重新跑质量门禁（Phase 4 → Phase 5）

### 常见错误速查

| 错误类型 | 表现 | 修正方法 | 参考 |
|---------|------|---------|------|
| AI 情绪词 | "一丝悲伤涌上心头" | 改为身体反应："喉咙发紧，指甲掐进掌心" | writing-craft.md §1 |
| 机械对话 | A："你怎么了？" B："我没事。" | 加入潜台词/权力博弈/情绪承接 | dialogue-mastery.md |
| 科普嘴 | 角色用整段对话讲设定 | 通过冲突/争论自然引出信息 | dialogue-mastery.md |
| 通篇同长度 | 每段都是 3-5 字碎句 | 按情绪 beat 合并，沉淀处长句，冲突处短句 | format-and-structure.md |
| 标点堆砌 | "！？……" 乱用 | 标点服务语气，犹豫用动作/短句/换行 | format-and-structure.md |
| 上帝视角 | "他不知道的是…" | 删除，改为角色感知范围内的表达 | anti-ai-writing.md |
| 温吞保守 | 爽点一笔带过 | 展开极端反应、围观哗然、代价反转 | emotional-methods.md |
| 章末假悬念 | "更大的风暴即将来临" | 删除，用动作/对话/具体事件制造悬念 | hooks-paragraph.md |
