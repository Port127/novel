## 一、项目定位

**Novel V2 是一个 AI 辅助小说写作工具。**

它不是用来"自动生成小说"的工具，而是用来**辅助作者完成创作流程**的交互式系统。

**类比**：建筑师不会让 AI 自动设计一栋建筑，而是让 AI 提供方案参考、检查规范、生成文档。本项目就是那个"AI 设计助手"。

**与 novel-material 的关系**：

| 项目 | 定位 | 关系 |
|------|------|------|
| novel-material | 素材检索库 | 上游，提供参考样例 |
| novel-v2 | 写作工具 | 本项目，调用上游检索服务 |

## 二、用户的两个写作场景

### 场景 A：前期规划阶段

用户在动笔之前，需要构建小说的整体框架。

**需要辅助生成的内容：**

| 内容 | 说明 | 生成方式 |
|------|------|---------|
| 世界观设定 | 力量体系、势力关系、地理空间 | 交互式询问 + AI 生成 |
| 大纲 | 全书结构（幕 → 序列 → 节拍） | 交互式询问 + AI 生成 |
| 人物设定 | 主角、反派、配角的人物档案 | 交互式询问 + AI 生成 |
| 章节规划 | 将大纲转化为章节摘要和张力曲线 | 大纲解析 + AI 生成 |

**用户工作流：**

```
1. 用户创建项目
   ↓
2. 与 Claude 讨论，明确核心想法
   ↓
3. Claude 逐步询问：核心设定 → 主角 → 冲突 → 结构
   ↓
4. Claude 生成初稿，展示给用户
   ↓
5. 用户确认或要求调整
   ↓
6. 状态流转：planning → drafting
```

### 场景 B：章节写作阶段

用户在写具体章节时，需要 AI 辅助生成正文。

**需要辅助生成的内容：**

| 内容 | 说明 | 生成方式 |
|------|------|---------|
| 章节正文 | 根据章纲生成完整正文 | AI 生成 |
| 续写 | 在已有正文基础上继续写作 | AI 续写 |
| 改写 | 润色、扩展、精简、重写 | AI 改写 |

**用户工作流：**

```
1. 用户选择要写的章节
   ↓
2. Claude 确认章节摘要和上下文
   ↓
3. Claude 询问写作方向（可选）
   ↓
4. Claude 生成正文
   ↓
5. 展示正文，询问是否满意
   ↓
6. 用户选择：接受 / 续写 / 改写
   ↓
7. 状态流转：planned → draft → written → revised
```

## 三、需求优先级

### 3.1 需求优先级

| 优先级 | 需求 | 理由 | 验收标准 |
|-------|------|------|---------|
| **P0** | 章节正文生成 | 场景 B 核心需求，写作时必用 | 生成正文 ≥ 1500 字，语义连贯 |
| **P0** | 大纲生成 | 场景 A 核心需求，规划阶段必用 | 生成完整大纲结构（幕 → 序列 → 节拍）|
| **P1** | 世界观生成 | 场景 A 核心需求 | 生成完整世界观设定 |
| **P1** | 人物生成 | 场景 A 核心需求 | 生成完整人物档案 |
| **P1** | 章节规划 | 大纲 → 章纲转化 | 每章有摘要 + 张力值 |
| **P1** | 续写/改写 | 正文写作辅助 | 续写 ≥ 指定字数，改写保留核心情节 |
| **P2** | 素材检索（调用 novel-material） | 参考辅助 | 检索结果可关联到设定 |
| **P2** | 导出（TXT/MD/EPUB） | 成果输出 | 导出文件格式正确 |
| **P2** | 统计查看 | 进度跟踪 | 显示字数、章数、状态 |

### 3.2 需求依赖关系（强制）

**核心原则**：小说应该"生长和演进"，而非"一次性生成"。

```
世界观（根基）
    ↓ 生长
人物（在世界观中生根）
    ↓ 生长
大纲（在人物弧线上生长）
    ↓ 生长
章节规划（在大纲中细化）
    ↓ 生长
正文（在所有设定基础上写作）
```

**强制依赖顺序**：

| 阶段 | 设定类型 | 前置依赖 | 完善度阈值 | 对应 Skill |
|------|---------|---------|-----------|-----------|
| 1 | 世界观 | 无 | 80% | nm → 交互生成 |
| 2 | 人物 | 世界观完善 | 70% | generate-character |
| 3 | 大纲 | 世界观+人物完善 | 85% | generate-outline |
| 4 | 章节规划 | 大纲完善 | 100% | generate-chapter |
| 5 | 正文 | 章节规划存在 | - | write-chapter |

**强制检查**：

Skills 必须检查前置完善度，未达标则阻止进入下一阶段。

完善度检查命令：
```bash
novel generate (内建检查) {project_id} {setting_type}
```

完善度标准：`data/schemas/completeness.schema.yaml`

**设计影响**：

- 项目管理是最底层依赖
- 大纲生成 → 章节规划 → 正文写作 是核心链路
- 续写/改写依赖已有正文

## 四、核心功能需求

### 需求 1：项目管理

**触发时机**：场景 A — 开始创作前

**用户意图**：创建新的写作项目，或管理已有项目

**CLI 命令**：

```bash
novel new "书名" --genre 类型 --author 作者
novel list
novel show <project_id>
novel delete <project_id>
```

**返回内容**：

- 项目 ID（系统生成）
- 项目目录结构
- 基本信息确认

**需求特征**：

| 特征 | 说明 | 设计影响 |
|------|------|---------|
| ID 自动生成 | 格式 `nv_{YYYYMMDD}_{random4}` | 用户无需手动命名 |
| 目录标准化 | 统一的目录结构 | 所有项目可互换处理 |
| 状态追踪 | planning → drafting → revising → completed | 状态机管理 |

---

### 需求 2：世界观生成

**触发时机**：场景 A — 前期规划

**用户意图**：构建小说的世界观设定

**交互式询问流程**：

```
1. 确认类型（修仙/都市/科幻/悬疑等）
   ↓
2. 询问力量体系（有无等级？如何晋升？）
   ↓
3. 询问势力关系（主要势力？冲突关系？）
   ↓
4. 询问地理空间（世界范围？重要地点？）
   ↓
5. 询问核心规则（世界运行的底层逻辑）
   ↓
6. 汇总确认，生成 YAML
```

**CLI 命令**：

```bash
novel generate world <project_id> [--prompt "描述"]
```

**返回内容**：

- `settings/worldbuilding.yaml`
- 包含：power_system, factions, locations, lore

---

### 需求 3：大纲生成

**触发时机**：场景 A — 前期规划

**用户意图**：规划全书结构

**交互式询问流程**：

```
1. 确认核心设定（一句话概括故事）
   ↓
2. 询问主角设定（名字、起点、终点）
   ↓
3. 询问冲突设计（外部冲突、内部冲突）
   ↓
4. 询问结构规划（章数、幕数）
   ↓
5. 汇总确认，生成大纲
```

**CLI 命令**：

```bash
novel generate outline <project_id> --chapters 50 [--from-draft 草稿]
```

**返回内容**：

- `settings/outline.yaml`
- 包含：premise, acts, plotlines, hooks, pacing_curve

---

### 需求 4：人物生成

**触发时机**：场景 A — 前期规划

**用户意图**：设计主角、反派、配角

**交互式询问流程**：

```
1. 询问主角设定
   ↓
2. 询问反派设定
   ↓
3. 询问关键配角
   ↓
4. 询问人物关系
   ↓
5. 汇总确认，生成人物档案
```

**CLI 命令**：

```bash
novel generate character <project_id> [--from-draft 草稿]
```

**返回内容**：

- `settings/characters.yaml`
- 包含：人物档案、关系图、弧线设计

---

### 需求 5：章节规划

**触发时机**：场景 A → 场景 B 过渡

**用户意图**：将大纲转化为章节摘要和张力曲线

**CLI 命令**：

```bash
novel generate chapter <project_id>
```

**返回内容**：

- `chapters/_index.yaml`
- 每章：摘要、张力值（1-5）、状态

---

### 需求 6：章节正文生成

**触发时机**：场景 B — 章节写作

**用户意图**：根据章纲生成完整正文

**交互式流程**：

```
1. 确认章节摘要和上下文
   ↓
2. 询问写作方向（可选）
   ↓
3. 生成正文
   ↓
4. 展示正文，询问是否满意
   ↓
5. 用户选择：接受 / 续写 / 改写
```

**CLI 命令**：

```bash
novel write new <project_id> <章节号> [--prompt "方向"]
novel write continue <project_id> <章节号> --length 1000
novel write revise <project_id> <章节号> --mode polish
```

**返回内容**：

- `chapters/chapter_{num}.md`
- 状态更新：planned → draft → written → revised

## 五、Skills 系统

Skills 是交互式创作的核心入口。

### Skill 设计原则

| 原则 | 说明 |
|------|------|
| 交互式询问 | 逐步确认，不跳过用户 |
| 尊重已有设定 | 不覆盖用户已确认的内容 |
| 结构化输出 | 生成 YAML/Markdown 格式 |
| 可调整性 | 生成后支持用户修改 |

### Skills 与 CLI 对应

| Skill | CLI 等价 | 交互式询问 |
|-------|----------|-----------|
| `generate-outline` | `generate.py outline` | 5 步询问流程 |
| `generate-character` | `generate.py character` | 4 步询问流程 |
| `generate-chapter` | `generate.py chapter` | 确认大纲后转化 |
| `write-chapter` | `write.py new/continue/revise` | 确认摘要 + 询问方向 |
| `show-project` | `project.py show` + `stats.py` | 直接展示 |
| `export-novel` | `export.py` | 直接导出 |

## 六、关键约束与原则

### 6.1 状态流转

**项目状态**：

```
planning → drafting → revising → completed
```

| 状态 | 允许操作 | 禁止操作 |
|------|---------|---------|
| planning | 生成设定、规划章节 | 写正文、导出 |
| drafting | 写正文、续写、改写 | 修改世界观/大纲 |
| revising | 改写、润色 | 新增章节 |
| completed | 导出、统计 | 任何修改 |

**章节状态**：

```
planned → draft → written → revised
```

| 状态 | 字数要求 | 允许操作 |
|------|---------|---------|
| planned | 0 | 写正文 |
| draft | < 1500 | 续写 |
| written | ≥ 1500 | 改写、润色 |
| revised | ≥ 1500 | 备份后修改 |

### 6.2 禁止操作

| 禁止 | 原因 |
|------|------|
| 跳过用户确认直接生成 | 用户必须参与创作流程 |
| 在未规划章节时写作正文 | 必须先有章纲 |
| 修改已 revised 状态章节时不创建备份 | 防止丢失已完成工作 |
| 覆盖已有设定 | 尊重用户已确认的内容 |

### 6.3 草稿系统

用户可通过多种方式提供初始想法：

| 来源 | 使用方式 |
|------|----------|
| `settings/notes.yaml` | 项目内草稿区，Skill 自动读取 |
| 直接提供内容 | `--from-draft "想法..."` |
| 外部文件 | `--from-draft path/to/file.txt` |

## 七、规模目标

| 指标 | 数值 |
|------|------|
| 单项目章节数 | 支持 100+ 章节 |
| 单章节字数 | 建议 1500-3000 字 |
| 设定文件大小 | 每个 < 100KB |
| LLM 响应时间 | < 30 秒（单章节生成）|
| 导出速度 | < 10 秒（100 章小说）|

## 八、不做的内容

| 不做内容 | 原因 |
|----------|------|
| 自动生成完整小说 | 用户必须参与创作流程 |
| 事件拆分（"戏剧动作"级别） | 边界不可控，投入产出比极低 |
| 精细张力曲线（逐章标注） | 无法检验；章节级张力（1-5）精度足够 |
| 模板化自动生成 | 用户需要的是"交互式创作"，不是"模板套用" |
| 多人协作 | 当前定位为单用户工具 |
| Web UI | 所有操作通过 Claude Code 和 CLI 完成 |
| 实时同步 | 本地工具，无需同步 |
| 版本管理 | Git 已覆盖 |

## 九、成功标准

本项目是否成功，取决于用户能否在实际写作中：

1. **交互式创作流畅**：询问 → 确认 → 生成 → 调整的流程顺畅
2. **设定一致性**：世界观/人物设定在整个写作过程中保持一致
3. **正文质量可接受**：生成正文语义连贯，符合章纲要求
4. **进度可控**：用户可随时查看项目状态和统计
5. **成果可导出**：最终作品可导出为 TXT/MD/EPUB

**可度量指标**：

| 指标 | 标准 |
|------|------|
| 正文生成成功率 | > 95% |
| 设定一致性检查 | 无冲突 |
| 用户调整次数 | 平均 < 3 次/章节 |
| 导出文件格式正确率 | 100% |

## 十、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| **LLM 调用失败** | 生成中断 | 指数退避重试 + 断点续传 |
| **LLM 输出格式错误** | YAML 解析失败 | 输出校验 + 降级处理 |
| **正文质量不稳定** | 用户不满意 | 多轮询问 + 改写选项 |
| **设定冲突** | 人物/世界观不一致 | 设定检查 + 覆盖保护 |
| **长小说性能下降** | 统计/导出缓慢 | 惰性加载 + 缓存 |
| **状态流转混乱** | 用户绕过流程 | 状态检查 + 操作禁止 |
