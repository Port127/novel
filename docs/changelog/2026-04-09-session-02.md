## 二、详细变更

### A. 评估闸门硬化（ADR-16）

**新增协议：** `_protocols/eval-gate.md`

核心设计：

- **结果落盘**：`chapter-review`、`voice-check`、`anti-ai-check` 的评估结果统一写入 `chapters/{chapter_id}_review.yaml`
- **评分维度**：structure（结构）、character（角色）、hook（伏笔）、voice（对白）、anti-ai（AI 痕迹）、continuity（连续性）
- **阈值阻断**：`pipeline-draft-polish` 在推进状态前检查分数，`anti-ai < 60` 或 `structure < 50` 时阻断推进
- **逃生通道**：`--force` 跳过闸门，`--threshold N` 调整阈值

**影响 Skill（4 个）：**
| Skill | 改动 |
|-------|------|
| `chapter-review` | 新增步骤 3：写入 `_review.yaml` 结构化结果 |
| `voice-check` | 新增步骤 6：写入 `_review.yaml` 对白评估 |
| `anti-ai-check` | 双写到 `_review.yaml`（原有输出保留） |
| `pipeline-draft-polish` | 新增步骤 7 闸门检查；新增 `--force` / `--threshold` 参数 |

### B. 机械化约束（ADR-17）

**新增 Skill：** `project-lint/SKILL.md`

7 大检查类别，每条错误消息内嵌修复命令：

| 类别 | 检查内容 | 典型错误消息 |
|------|----------|-------------|
| 章节文件-索引一致性 | index.yaml 条目 vs chapters/*.md 文件 | `ch003 在 index.yaml 中但文件不存在 → /chapter-create ch003` |
| 角色文件-索引一致性 | character_index.yaml vs characters/*.yaml + 五件套检查 | `张三缺少 fatal_flaw → /character-edit 张三 补充致命缺陷` |
| 设定文件-索引一致性 | worldbuilding.yaml vs entries/*.yaml + 过期/tentative 检测 | `rule_001 tentative 超 10 章 → /setting-edit rule_001 --status confirmed` |
| 钩子健康度 | 逾期 major/minor、无截止日、引用幽灵章节 | `f001 major 逾期 5 章 → /hook-resolve f001 --recover ch015` |
| 操作日志健康度 | 遗留 in_progress、日志过大 | `ops_log.yaml 有 3 个 in_progress → /project-reindex 清理` |
| 交叉引用完整性 | 关系/大纲/时间线引用幽灵角色 | `timeline 引用已删除角色"李四" → /character-add 李四 重建或清理引用` |
| 命名规范 | 禁止模式、同音混淆 | `角色名"梓萱"命中 forbidden_patterns → 建议替换` |

**参数：**
- `--scope [category]`：按领域筛选（只检查钩子/只检查角色等）
- `--fix`：安全自动修复（仅修复明确无争议的问题，如 orphaned index entries）
- `--summary`：只看汇总统计，不看详情

### C. 上下文预算控制（ADR-18）

**新增协议：** `_protocols/context-budget.md`

三级读取策略：

| 级别 | 策略 | 触发场景 |
|------|------|---------|
| Level 1 索引优先 | 只读索引/摘要文件（index.yaml, outline.yaml, character_index.yaml） | 所有 skill 默认级别 |
| Level 2 聚焦读取 | 针对性读取特定角色的 yaml、特定设定条目、特定章节 | 已知操作对象时（如编辑某角色） |
| Level 3 全量扫描 | 读取所有相关文件（glob *.yaml） | 显式触发：`/project-reindex`、`/project-lint --scope all` |

**关键规则：**
- 禁止 Level 3 读取超过 20 个文件
- `consistency-check` 分三轮加载（索引 → 可疑项 → 语义比对），不一次读全部
- `chapter-draft` 按 P0-P4 优先级分配上下文，15000 字 token 时从 P4 开始裁剪
- 高频 skill 的 SKILL.md 中标注每步的"上下文成本"

**影响 Skill（2 个）：**
| Skill | 改动 |
|-------|------|
| `chapter-draft` | 步骤 1 按 P0-P4 优先级收集上下文（P0 必选 → P1 强烈建议 → P2 建议 → P3 可选 → P4 锦上添花） |
| `consistency-check` | 步骤 1 分三轮加载：索引层 → 可疑项层 → 语义比对层 |

### D. 知识新鲜度检查（ADR-19）

**增强 Skill：** `novel-doctor`

新增 §5 知识新鲜度，5 个子检查：

| 子检查 | 检测内容 | 阈值 |
|--------|----------|------|
| 5a 设定新鲜度 | tentative 超 N 章未 confirm、过期设定无后继、过期设定仍被引用 | N=5 章 |
| 5b 角色新鲜度 | 角色超 N 章未出场、current_state 过时、五件套未填 | N=8 章 |
| 5c 钩子新鲜度 | major/minor 逾期、无截止日、单章钩子密度异常 | major 逾期=0 |
| 5d 大纲新鲜度 | 未写章节堆积（idea 状态超 10 个）、偏离未同步 | idea ≤ 10 |
| 5e 文档新鲜度 | PROJECT_MAP 过旧、风格模板过旧（超 30 章未更新） | 30 章 |

**输出格式：** 按严重程度分组（Critical / Warning / Info），与现有诊断维度（文件完整性、时间线、角色一致性）并列输出。

### E. 反馈循环闭合（ADR-20）

这是 A（评估闸门）的闭环侧——审查结果不再是用完就丢的终端输出。

**闭环路径：**

```
chapter-draft 生成草稿
    ↓
pipeline-draft-polish 审查（chapter-review + voice-check + anti-ai-check）
    ↓
结果写入 chapters/{chapter_id}_review.yaml
    ↓
下次 chapter-draft 读取 _review.yaml
    ↓
注入写作指令："前次审查发现以下问题，本次生成应避免..."
    ↓
生成结果自动改进
```

**实现细节：**

1. `_review.yaml` 格式：
   ```yaml
   chapter_id: ch003
   date: 2026-04-09
   scores:
     structure: 72
     character: 85
     hook: 68
     voice: 90
     anti_ai: 55
     continuity: 80
   blocking_issues: []         # 阻止状态推进的问题
   suggestions:                # 按优先级排序的建议
     - priority: high
       category: anti_ai
       issue: "开头三段描写过于抽象"
       suggestion: "用具体动作和对话替代抽象描写"
   resolved_issues: []         # 已解决的问题追踪
   ```

2. `chapter-draft` 步骤 1 新增 P0 必读项"前次审查反馈"：
   - 读取 `_review.yaml` 中的 `blocking_issues` 和 `high` priority 的 `suggestions`
   - 注入到写作指令中
   - 生成后在写作备忘中标注已参考的反馈

## 四、架构决策：文件 vs 数据库（暂不实施）

用户提出"纯 skill + 文本是否应该加数据库提升稳定性"的问题，经过完整分析后达成以下结论：

### 当前不稳定根源分析

| 不稳定类型 | 频率 | 数据库能解决吗 |
|-----------|------|--------------|
| 手动删文件导致孤儿引用 | 中 | **能** — 外键约束 / 引用完整性 |
| 多 skill 写同一文件中断导致半写 | 低 | **能** — 事务原子性 |
| AI 偷偷混入非来源内容 | 高 | **不能** — prompt 遵从问题 |
| AI 把别称换成正式名 | 高 | **不能** — 生成逻辑问题 |
| 起名不自然 | 中 | **不能** — 风格约束问题 |
| 大纲塞太满 | 中 | **不能** — 粒度判断问题 |

**结论**：当前系统最大的不稳定来源是 AI 行为，不是数据完整性。我们刚做的 6 项架构加固（预检、操作日志、名字解析等）已经在 skill 层面补了数据完整性的短板。

### 加数据库会破坏什么

1. **人可以直接改文件** — 核心工作流。加了数据库后必须通过 API 或 skill 改，否则数据库和文件脱节
2. **Git 友好性** — 现在 `git diff` 可读，数据库二进制文件不可读
3. **零依赖** — 不需要启动服务、不需要安装数据库
4. **AI 上下文加载** — AI 直接 Read 文件即可获得完整上下文，走数据库多一层抽象多一层出错可能

### 未来的轻量级混合方案（触发条件达成时实施）

```
文件（保持不变）          索引层（新增，可选）
─────────────          ──────────────────
chapters/*.md          SQLite: chapters 表
characters/*.yaml  ←→  SQLite: characters 表
worldbuilding/         SQLite: settings 表
plot/outline.yaml      SQLite: hooks 表、cross_refs 表

写入路径：skill → 写文件 → 同步到 SQLite
查询路径：skill → 读 SQLite（快） → 需要全文时读文件
人工编辑：直接改文件 → /project-reindex 刷新 SQLite
```

**优点**：文件仍是 single source of truth，SQLite 只是加速索引，丢了可从文件重建，不需要服务器（嵌入式），Git 可 `.gitignore` 掉。

### 触发条件

- 单个项目超过 100 章，查询开始变慢
- 需要跨项目查询（"哪些书里有时间回溯机制"）
- 希望有 Web UI 做可视化看板

**当前决策**：保持纯文件架构，等项目规模到了再上索引层。

## 关键设计决策

### 为什么评估结果是 YAML 而不是 Markdown？

YAML 是结构化数据，可被下游 skill 程序化读取（`blocking_issues` 数组、`scores` 数值）。Markdown 是给人类看的报告，YAML 是给机器看的反馈——两者并存。

### 为什么 project-lint 不做成 Python 脚本而是做成 Skill？

本项目是 Skill Package（非代码仓库），没有 Python 运行时。linter 的本质也是"读取文件 → 检查规则 → 输出报告"，完全在 AI 的能力范围内。把规则编码成 Skill 的协议文件（而非 Python 代码），更符合项目架构。

### 为什么阈值默认 anti-ai < 60 和 structure < 50？

这两个是最低质量底线。anti-ai 低于 60 说明 AI 痕迹严重，structure 低于 50 说明章节缺乏基本骨架。低于这两个值推进状态相当于把半成品推到下游，污染后续所有流程。
