# 🔍 Novel V2 项目全面审视报告

> **视角**：一个想用这个工具写出能在起点/番茄上赚钱的小说的作者
> **调研范围**：AI 写作工具市场、网文行业生态、读者社区反馈、项目代码分析

---

## 一、结论先行：核心问题

> [!CAUTION]
> **本项目目前是一个「架构精美但无法上路的车」**——目录结构、Schema 设计、Pipeline 文档都很专业，但作为一个想赚钱的网文作者，**几乎没有一个功能是能直接用来写小说的**。

| 维度 | 评分 | 说明 |
|------|------|------|
| 项目管理 | ⭐⭐⭐ | 创建/列表/导出基本可用 |
| AI 生成能力 | ⭐ | `generate.py` 和 `write.py` **根本不存在** |
| 写作效率支撑 | ⭐ | 无日产量追踪、无续写、无批量生成 |
| 设定一致性管理 | ⭐ | 无校验、无交叉引用、无冲突检测 |
| 商业化适配 | ⭐ | 无平台格式、无更新节奏管理、无读者分析 |
| 数据完整性 | ⭐⭐ | outline 充实，但 characters/worldbuilding 基本空白 |

## 二、网文行业真实需求 vs 本项目能力对比

### 2.1 网文作者的生存现实

网文行业有一句话叫**「断更 = 死」**。一个想赚钱的网文作者面临的核心挑战：

| 挑战 | 行业标准 | 本项目支持 |
|------|----------|------------|
| **日更字数** | 4,000-6,000 字/天（2-3 章） | ❌ 无任何产能辅助 |
| **更新频率** | 每日不间断，停更 2-3 天读者就流失 | ❌ 无更新计划/节奏管理 |
| **黄金三章** | 前 3 章决定 90% 读者去留 | ❌ 无开篇优化工具 |
| **爽点节奏** | 每 3-5 章小爽点，15-20 章大爽点 | ❌ 无爽点/张力曲线规划 |
| **设定一致性** | 200+ 章不能出现矛盾 | ❌ 无一致性校验 |
| **字数规模** | 100-300 万字（200-600 章） | ⚠️ 规划了 200 章但无法执行 |
| **平台适配** | 起点/番茄格式要求 | ❌ 无平台格式导出 |

### 2.2 竞品对比

| 功能 | Sudowrite | NovelAI | 本项目 |
|------|-----------|---------|--------|
| AI 续写正文 | ✅ 核心功能 | ✅ 核心功能 | ❌ write.py 不存在 |
| 风格学习/模仿 | ✅ | ✅ 微调模型 | ❌ |
| 长篇记忆/一致性 | ⚠️ 有限 | ⚠️ lorebook | ❌ |
| 大纲生成 | ✅ | ❌ | ⚠️ 有设计无实现 |
| 人物管理 | ✅ | ✅ | ❌ characters.yaml 为空 |
| 多格式导出 | ✅ | ❌ | ✅ TXT/MD/EPUB |
| 项目管理 | ❌ | ❌ | ✅ CLI 可用 |
| 中文网文支持 | ❌ | ❌ | ⚠️ 定位正确但未落地 |

> [!IMPORTANT]
> **本项目最大的差异化优势**应该是「中文网文专属 AI 写作工具」——市面上没有一个工具真正理解网文的爽点节奏、修仙设定、升级体系。这是蓝海，但目前完全没有落地。

## 三、逐项深挖：具体问题清单

### 3.1 🚨 致命问题（没有这些，项目无法使用）

#### P0-1：核心生成脚本不存在

README 中宣传的 `generate.py` 和 `write.py` **完全不存在**：

```bash
# README 说可以这样用，但这些文件根本不存在：
python scripts/generate.py outline <project_id>   # ❌ 文件不存在
python scripts/write.py new <project_id> 1         # ❌ 文件不存在
python scripts/write.py continue <project_id> 1    # ❌ 文件不存在
python scripts/write.py revise <project_id> 1      # ❌ 文件不存在
```

这意味着项目的**核心价值——AI 辅助生成内容——完全无法工作**。

#### P0-2：LLM 集成不完整

- `utils/` 目录下只有 `completeness_check.py`，README 提到的 `llm_client.py` **不存在**
- `.env` 配置了 LLM API 相关变量，但没有代码使用它们
- 没有任何 prompt engineering 相关代码

#### P0-3：项目模板与实际数据结构不匹配

模板定义了精细的模块化目录结构：
```
settings/worldbuilding/power_system.yaml
settings/characters/protagonist/protagonist.yaml
settings/outline/acts/act_*.yaml
```

但实际项目使用的是扁平结构：
```
settings/outline.yaml        # 一个大文件
settings/characters.yaml     # 空的
settings/worldbuilding.yaml  # 几乎空的
```

这导致模板设计形同虚设。

### 3.2 ⚠️ 严重问题（影响写作效率和质量）

#### P1-1：无设定一致性管理

这是网文作者**最头疼的问题之一**。200+ 章的小说中：
- 角色名字写错（"张三" 突然变 "张山"）
- 修仙境界体系前后矛盾
- 地名/门派名不一致
- 角色性格 OOC（Out of Character）
- 时间线矛盾

**读者社区（龙的天空、百度贴吧）最常见的差评就是「设定崩了」**。

本项目虽然设计了 Schema，但：
- Schema 目录存在但内容未确认是否完整
- `completeness_check.py` 只检查文件是否存在，不检查内容一致性
- 没有交叉引用验证（比如大纲中提到的角色是否在 characters 中定义）

#### P1-2：无写作效率工具

对网文作者来说，效率就是生命：
- ❌ 没有日产量追踪（今天写了多少字？）
- ❌ 没有写作目标设定（日目标 / 周目标）
- ❌ 没有连更天数统计
- ❌ 没有写作速度分析
- ❌ 没有产能趋势图表

#### P1-3：无章节场景细分

当前 outline 只到「章节摘要」粒度，但实际写作需要：
- **场景分解**（Beat Sheet）：每章 3-5 个场景
- **情感节拍**：每个场景的情感走向
- **对话/描写/叙述比例**：控制节奏
- **字数预估**：每场景的目标字数
- **爽点标记**：哪里是本章的爽点

#### P1-4：无上下文管理机制

AI 生成长篇小说最大的痛点是**上下文窗口限制**。需要：
- 「前情提要」自动摘要：前 N 章的关键信息压缩
- 角色状态追踪：当前每个角色在哪里、什么状态、什么境界
- 伏笔管理：哪些伏笔已埋、哪些已回收
- 世界状态：当前世界发生了什么大事件

### 3.3 📋 重要缺失（影响商业化）

#### P2-1：无平台适配

不同平台有不同要求：

| 平台 | 章节字数 | 格式要求 | 签约门槛 |
|------|----------|----------|----------|
| 起点中文网 | 2,000-3,000 字/章 | 特定排版格式 | 3-5 万字后申请 |
| 番茄小说 | 1,500-2,500 字/章 | 更宽松 | 较低 |
| 晋江文学城 | 灵活 | 独有格式 | 按完结度 |

本项目的导出功能（TXT/MD/EPUB）都不是网文平台需要的格式。

#### P2-2：无读者反馈循环

成功的网文作者每天都要：
- 看评论区反应，调整后续剧情
- 分析哪些章节订阅/收藏变化大
- 根据读者投票决定剧情走向

项目完全没有这类功能的设计。

#### P2-3：无质量检测

网文读者最讨厌的「毒点」：
- **水文**：重复描述、无意义对话、凑字数
- **节奏拖沓**：3 章能讲完的事写了 10 章
- **套路重复**：同一种冲突-解决模式反复使用

应该有自动化检测：
- 重复率检测（相似句子/段落）
- 节奏分析（事件密度、对话/描写比例）
- 套路检测（是否陷入模式化）

## 五、从「写出能卖钱的小说」角度的优先建议

### 🎯 第一优先级：让工具能用起来

| # | 任务 | 理由 |
|---|------|------|
| 1 | **实现 `generate.py`** | 核心生成能力，没有它项目毫无意义 |
| 2 | **实现 `write.py`** | 正文写作是最终产出 |
| 3 | **实现 `llm_client.py`** | LLM 集成基础设施 |
| 4 | **补全 characters 和 worldbuilding** | 没有设定就无法生成一致的内容 |

### 🎯 第二优先级：解决网文作者核心痛点

| # | 任务 | 理由 |
|---|------|------|
| 5 | **上下文管理系统** | 长篇小说一致性的生命线 |
| 6 | **设定一致性校验** | 避免「设定崩了」的差评 |
| 7 | **日产量追踪 + 目标管理** | 维持更新频率 |
| 8 | **场景级章节规划** | 提升写作效率和质量 |

### 🎯 第三优先级：商业化闭环

| # | 任务 | 理由 |
|---|------|------|
| 9 | **平台格式导出** | 直接发布到起点/番茄 |
| 10 | **爽点/毒点检测** | 质量保证 |
| 11 | **水文检测** | 读者最反感的问题 |
| 12 | **进度仪表盘** | 可视化管理长期项目 |

## 六、行业洞察：本项目的机会

> [!TIP]
> **市面上没有一款工具真正为中文网文作者设计**。Sudowrite/NovelAI 面向英文文学，文心一言/通义千问是通用 AI。一个**深度理解网文套路、爽点节奏、修仙体系、升级设定**的 AI 写作工具，是真正的蓝海。

具体的差异化方向：
1. **爽点编排引擎**：自动规划每 N 章的爽点分布，确保读者「停不下来」
2. **网文体裁模板**：修仙、都市、系统流、重生流等各有一套模板和设定库
3. **境界/等级自动管理**：修仙小说的境界体系、都市小说的实力等级自动追踪
4. **反派智商保证**：AI 检测反派是否过于弱智化（读者大忌）
5. **伏笔追踪器**：埋下的伏笔自动提醒回收
6. **日更助手**：基于大纲自动生成当日章节草稿，作者只需润色

---

## 七、总结

**一句话评价**：项目的**设计理念和架构**是对的，但**离「让作者能写出赚钱小说」的目标差距巨大**——主要是因为核心生成功能完全没有实现，以及缺少对网文行业真实需求（日更压力、一致性管理、爽点节奏、平台适配）的深度支持。

当前状态更像是一个**项目管理工具的骨架**，而非**写作助手**。要实现项目愿景，需要从「用户每天打开这个工具写 4000-6000 字并发布到平台」的场景出发，重新审视每个功能的优先级。
