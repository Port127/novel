# 任务

读取用户的混合笔记文件，自动识别内容类型并分拣，经用户确认后派发到对应 skill 执行落地。

核心原则：**分拣展示 → 用户确认 → 才执行写入。绝不静默写入。**

## 前置检查

1. 读取 `.current.yaml` 获取 `current_path`
2. 确认 `$0`（文件路径）存在且可读
3. 如果 `$0` 未提供——检查用户当前打开/引用的文件，若是非项目结构文件（非 `chapters/`、`characters/`、`plot/` 等已有目录下的文件），将其作为输入
4. 若仍无法确定，提示用户指定文件

## 输入参数

- `$0` (file_path): 笔记文件路径（必需）
- `--quick`: 跳过逐条确认，所有识别到的条目按默认类型直接落地（设定标记 tentative，角色标记基础档案）
- `--dry-run`: 只做分拣展示，不执行任何写入

## 执行步骤

### 1. 全文阅读与分段

读取文件全文，按自然段落或语义边界切分为独立条目。

分段规则：
- 空行分隔的段落视为独立条目
- 同一段落内如果涉及多个不同类型的内容，进一步拆分
- 连续描述同一事物的多个段落合并为一条
- 列表项（`-`、`*`、`1.`）每项视为独立条目

### 2. 智能分类

对每条内容判断类型：

| 类型 | 识别信号 | 派发目标 |
|------|---------|---------|
| `setting` | 世界规则、力量体系、限制条件、地理、术语、组织势力 | `/setting-add` |
| `character` | 人物描述、姓名+身份、性格、背景、外貌 | `/character-add` |
| `plot` | 事件、冲突、转折、"第X章"、剧情走向 | `/plot-add` |
| `timeline` | 时间点、"第X天"、日期、事件先后顺序 | `/timeline-add` |
| `relationship` | 两人关系、A对B、情感变化、盟友/敌对 | `/relationship-add` |
| `worldbuilding_narrative` | 大段世界观描述、历史、文化背景叙述 | 追加到 `setting.md` |
| `meta` | 写作计划、TODO、"接下来要写"、"记得改" | 不落地，仅展示提醒 |
| `unclear` | 无法归类 | 由用户指定 |

分类时的原则：
- 一条内容只归一个主类型，不做多重标记
- 优先从内容语义判断，而非关键词匹配
- 模棱两可时标记为 `unclear`，交给用户决定

### 3. 分拣展示

按类型分组展示，每条附带原文摘要和推断理由：

```
📋 笔记分拣结果：{{file_name}}

共识别 {{total}} 条内容：

━━━ 🌍 设定（{{count}}条）━━━
  1. [setting] 污染能量不能储存，必须即时消耗
     → 推断类别：power_system
  2. [setting] 灵脉枢纽城市只有七座
     → 推断类别：geography

━━━ 👤 角色（{{count}}条）━━━
  3. [character] 林砚，19岁学徒，温顺但极度怕被抛下
     → 推断定位：supporting
  4. [character] 宗门长老赵无极，老谋深算，暗中控制三条商路
     → 推断定位：antagonist

━━━ 📖 剧情（{{count}}条）━━━
  5. [plot] 第七章主角在废矿发现旧日记，揭露宗门灭门真相
     → 推断位置：第7章

━━━ ⏰ 时间线（{{count}}条）━━━
  6. [timeline] 第五天黎明，北城沦陷
     → 推断关联：ch008

━━━ 🔗 关系（{{count}}条）━━━
  7. [relationship] 张三和李四因误会彻底决裂
     → 推断阶段：ch010 之后

━━━ 📝 备忘（{{count}}条，不落地）━━━
  8. [meta] 记得第三章的伏笔要在第十章回收

━━━ ❓ 待定（{{count}}条）━━━
  9. [unclear] 这段文字无法确定类型，请指定

请确认分拣结果：
  - 回复 OK 全部确认
  - 回复编号+新类型 调整（如"3 改 plot"）
  - 回复编号+skip 跳过（如"8 skip"）
  - 回复编号+合并编号 合并条目（如"1 合并 2"）
```

### 4. 用户确认与调整

等待用户确认。用户可以：
- 全部确认
- 调整单条类型
- 跳过某些条目
- 合并条目
- 拆分条目
- 补充信息

**用户确认前不执行任何写入。**

### 5. 批量派发

确认后，按类型依次派发到对应 skill：

1. **设定类** → 调用 `/setting-add`
   - 自动推断 `category`
   - 默认 `status: tentative`
   - `source` 标注为笔记文件路径和条目编号

2. **角色类** → 调用 `/character-add`
   - 从描述中提取姓名、定位、年龄、身份、性格
   - 不足的字段留空，后续通过 `/character-edit` 补充

3. **剧情类** → 调用 `/plot-add`
   - 推断章节位置
   - 提取事件、冲突、转折

4. **时间线类** → 调用 `/timeline-add`
   - 提取时间点、事件、关联角色

5. **关系类** → 调用 `/relationship-add` 或 `/relationship-log`
   - 首次建立关系用 `add`，关系变化事件用 `log`

6. **世界观叙述类** → 追加到 `worldbuilding/setting.md`

7. **备忘类** → 不写入，仅在最终报告中列出提醒

8. **待定类** → 跳过（用户已在确认阶段处理）

派发过程中如果某条写入失败（如同名角色已存在），记录失败原因，继续处理下一条。

### 6. 更新状态

更新 `{current_path}/.novel/state.yaml`：
- `project.updated`：今天日期
- 各计数字段按实际写入结果更新

### 7. 输出汇总报告

```
✅ 笔记分拣完成：{{file_name}}

📊 处理结果：
   🌍 设定：{{n}} 条已写入（tentative）
   👤 角色：{{n}} 条已创建
   📖 剧情：{{n}} 个节点已添加
   ⏰ 时间线：{{n}} 条已记录
   🔗 关系：{{n}} 条已建立/记录
   📝 备忘：{{n}} 条（未落地）
   ⏭️ 跳过：{{n}} 条
   ❌ 失败：{{n}} 条

{{#if failures}}
⚠️ 失败条目：
{{failures_detail}}
{{/if}}

{{#if memos}}
📌 备忘提醒：
{{memos_list}}
{{/if}}

下一步：
   /pipeline-setting-consolidate    整固 tentative 设定
   /character-edit [角色名]          补全角色档案
   /consistency-check               检查一致性
```

## 注意事项

- **与 `draft-ingest` 的边界**：如果文件包含连贯叙事（有人物在做事、事件在推进），应建议用户对该段落使用 `/draft-ingest` 做深度消化，而非直接分拣。`note-triage` 处理的是碎片化笔记，不做故事 DNA 分析。
- 分拣阶段是**展示+确认**，不是自动执行。用户有完全控制权。
- 一条笔记只归一个类型。如果一段话同时提到角色和设定，按主要内容归类，必要时拆分。
- `--quick` 模式适合灵感爆发后快速落地，但所有设定都是 tentative，需要后续整固。
- 大段世界观叙述不要拆成碎片条目，保持叙述完整性追加到 `setting.md`。
- 如果笔记内容量很大（>50 条），分批处理，每批 10-15 条，避免确认疲劳。
- 派发失败不阻塞整体流程，记录后继续。
