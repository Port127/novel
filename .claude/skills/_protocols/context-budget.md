# 上下文预算协议（Context Budget Protocol）

> 受 Anthropic 上下文工程文章启发：上下文是有限资源，存在 context rot。  
> "找到最小的高信号 token 集合，最大化期望结果的概率。"

## 核心问题

Skill 执行时读取大量文件，导致：
1. **信息过载**：AI 的注意力被无关内容稀释，关键约束被忽略
2. **Context rot**：上下文越长，模型回忆准确率越低
3. **不稳定**：读太多文件时，AI 行为变得不可预测

## 三级读取策略

### Level 1：索引优先（默认）

先读索引/摘要，只在发现需要时才深入读原文。

**适用场景**：收集上下文阶段（chapter-draft 的步骤 1、consistency-check 的步骤 1）

**操作方式**：
- 读 `chapters/index.yaml` 的 `summary` 字段，而非打开每个 `.md`
- 读 `character_index.yaml` 的摘要信息，只对本章出场角色读完整 `.yaml`
- 读 `worldbuilding.yaml` 的条目列表，只对本章依赖的设定读完整 `entries/*.yaml`

### Level 2：聚焦读取

只读与当前操作直接相关的文件。

**适用场景**：针对性操作（chapter-review、voice-check、anti-ai-check）

**操作方式**：
- `chapter-review` 只读目标章节 + 前后章 summary（不读全部章节）
- `voice-check` 只读目标角色的对白 + 对手角色对白（不读全部角色卡）
- `anti-ai-check` 只读目标章节 + 规则文件（不读设定和大纲）

### Level 3：全量扫描（显式触发）

读取所有相关文件。

**适用场景**：仅在用户显式运行全量检查时（consistency-check、project-reindex）

**操作方式**：正常读取所有文件，但仍遵循"索引先行"——先读索引确定范围，再读原文。

## 实践指南

### chapter-draft 的上下文预算

按优先级分配上下文（从高到低裁剪）：

| 优先级 | 内容 | 估计 token | 可裁剪性 |
|--------|------|-----------|---------|
| P0 | 前情摘要链（所有前章 summary） | 低 | 不可裁剪 |
| P0 | 本章出场角色 current_state | 低 | 不可裁剪 |
| P0 | 前次审查反馈（blocking_issues） | 低 | 不可裁剪 |
| P1 | 前章结尾段落 | 中 | 可缩短到最后 300 字 |
| P1 | 本章场景大纲 | 中 | 不可裁剪 |
| P2 | 出场角色完整档案 | 高 | 可只取 speech_pattern + traits + fatal_flaw |
| P2 | 关系状态和近期事件 | 中 | 可只取当前状态，跳过历史事件 |
| P3 | 世界观设定 | 高 | 可只取本章引用的条目 |
| P3 | 大纲全文 | 高 | 可只取本章节点 |
| P4 | 时间线全文 | 中 | 可只取本章时间段 |
| P4 | 素材库检索结果 | 中 | 可跳过 |

**裁剪规则**：当上下文总量预估超过 15000 字时，从 P4 开始向上裁剪。

### consistency-check 的上下文预算

由于一致性检查需要全量数据，采用"分阶段加载"策略：

1. **第一轮**：只读索引文件（index.yaml、character_index.yaml、worldbuilding.yaml、outline.yaml），完成可从索引判断的检查（文件一致性、状态合法性、钩子统计）
2. **第二轮**：对第一轮发现的可疑项，加载具体文件深入检查
3. **第三轮**：对已写章节做语义比对时，只读 summary，不读全文（除非 summary 为空）

## Skill 中的标注方式

在 SKILL.md 的读取步骤中，用以下标记说明上下文策略：

- `[索引优先]`：先读索引，按需深入
- `[聚焦读取]`：只读与当前操作直接相关的文件
- `[必读]`：无论预算如何都不可跳过
- `[可裁剪]`：上下文紧张时可缩短或跳过
