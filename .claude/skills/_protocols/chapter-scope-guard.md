# 章节容量守卫协议（Chapter Scope Guard）

> 适用于 `chapter-draft`、`pipeline-chapter-kickoff` 等涉及章节内容规划和生成的 skill。

## 核心原则

**一章只写一章的事。剧情弧线可以跨多章，但每章只推进弧线的一个阶段。**

用户描述一段剧情时，这段剧情可能天然需要 2-5 章来承载。AI 不能把所有内容压缩进一章，也不能擅自替用户决定怎么拆分。

## 容量信号检测

以下任一信号出现，说明当前章节的内容规划可能超载：

| 信号 | 判断依据 |
|---|---|
| 场景数过多 | 场景大纲中有 4 个以上独立场景（不同时间/地点/人物组合） |
| 事件密度过高 | 一章中包含 2 个以上重大转折（每个都值得独立铺垫和消化） |
| 情绪跨度过大 | 从日常到高潮到收尾全在一章完成，缺少呼吸空间 |
| 字数预估超标 | 按场景大纲估算，预计字数超过目标字数的 1.5 倍 |
| 多线并行 | 同一章切换 3 条以上叙事线，每条都有推进 |
| 时间跨度过长 | 本章内时间跨度超过 3 天，且不是蒙太奇/过渡章 |

## 触发时的行为

### 在 chapter-draft 中

当检测到容量超载信号时，**不生成初稿**，而是：

1. **停下来，输出拆章建议**：

```
⚠️ 章节容量预警：本章场景大纲内容可能超出单章承载量

📋 当前章节：{{chapter_id}}
📊 检测到的信号：
   - {{信号1}}
   - {{信号2}}

📖 建议拆分方案：

方案 A：按场景断点拆分
  {{chapter_id}}：{{场景1-2 摘要}}（约 {{N}} 字）
  {{next_chapter_id}}：{{场景3-4 摘要}}（约 {{N}} 字）

方案 B：按情绪节奏拆分
  {{chapter_id}}：{{蓄力+爆发 摘要}}（约 {{N}} 字）
  {{next_chapter_id}}：{{余震+新钩子 摘要}}（约 {{N}} 字）

请选择：
  A / B / 自定义拆法 / 不拆（确认全部写入本章）
```

2. **用户选择后才继续**：
   - 选择拆分 → 更新当前章节的场景大纲（只保留本章部分），创建下一章并写入剩余场景大纲
   - 选择不拆 → 在写作备忘中标记"用户确认高密度章节"，正常生成

### 在 pipeline-chapter-kickoff 中

当用户给出的章节目标（goal）包含多个明显阶段时：

1. **识别弧线阶段**：分析目标描述中的事件序列，拆分出独立阶段
2. **输出多章规划建议**：

```
📖 检测到本段剧情可能跨越多章

你的目标：{{用户原始目标}}

建议章节规划：

  {{chapter_id_1}}：{{阶段1 摘要}}
    - 核心事件：{{event}}
    - 结尾钩子：{{hook}}

  {{chapter_id_2}}：{{阶段2 摘要}}
    - 核心事件：{{event}}
    - 结尾钩子：{{hook}}

  {{chapter_id_3}}：{{阶段3 摘要}}
    - 核心事件：{{event}}
    - 结尾钩子：{{hook}}

请确认：
  1. 按建议创建 {{N}} 章
  2. 只创建第一章，后续再规划
  3. 全部放一章（我确认这一章就是要这么多内容）
  4. 自己指定拆法
```

3. **用户确认后才执行创建**

## 多章弧线的跨章衔接

当确认拆分为多章时，每章的场景大纲末尾必须包含：

```markdown
**本章在弧线中的位置**：{{弧线名称}} 的第 {{N}}/{{Total}} 章
**上一章留下的钩子**：{{hook_from_previous}}（无则写"弧线起点"）
**本章结尾交接**：{{交接给下一章的悬念/状态}}
```

## 行为约束

- **不替用户决定拆法。** 拆分方案是建议，不是命令。
- **每次只生成一章的初稿。** 即使规划了 3 章，也逐章推进，不批量生成。
- **拆章后不丢内容。** 从当前章移出的场景必须明确落入下一章的场景大纲，不能悬空。
- **尊重用户的"不拆"决定。** 用户确认高密度章节后，不再反复提醒。
