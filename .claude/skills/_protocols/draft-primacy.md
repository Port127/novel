# 草稿优先原则（Draft Primacy Protocol）

> 本协议适用于所有涉及草稿与结构化材料（大纲、人物卡、设定条目、时间线）的 skill。

## 核心原则

**草稿是真相来源，大纲等结构化材料是辅助工具。**

作者写下的正文代表其最新意图。当草稿与大纲/人物卡/设定/时间线发生冲突时，**默认方向是大纲等材料需要更新以匹配草稿，而不是草稿需要修复以匹配大纲。**

## Skill 行为准则

### 检测到冲突时

1. **停下，报告，不要自行解决。**
2. 输出冲突清单，格式见下方模板。
3. 等待用户显式指令，才能更新任何结构化材料。

### 提取草稿内容时的来源边界

从草稿/源文件提取内容时，**每一条提取结果必须能指向源文件的具体行号**。

以下是明确的反模式，一旦发生就是系统性错误：

> **反模式示例**：
> 提取草稿情节点，列出"宿舍日常 → 室友失控 → 灰域展开 → 逃亡 → 被救 → **团队发现**"。
> 其中"团队发现"来自 `outline.md`，源文件没有这行。
> 这属于把非来源材料悄悄混入提取结果，违反来源边界规则。

每条提取内容必须打来源标签（见 `_protocols/from-extraction.md`）：
- `✅ 原文`（附行号）→ 可直接列入
- `🔶 推断`（从上下文合理推断）→ 列入但必须标注，等用户确认
- `❌ 非来源`（来自 outline/角色卡/设定等其他材料）→ **禁止混入提取结果**

如需展示其他材料的内容作对照，在输出末尾**单独列一节**，明确标注"以下来自 [文件名]，非本次草稿内容"。

### 永远不自动执行的操作

- 根据草稿内容更新 `plot/outline.md`
- 根据草稿行为修改 `characters/*.yaml`
- 根据草稿规则修改 `worldbuilding/entries/*.yaml`
- 根据草稿事件更新 `timeline/main.yaml`
- 根据草稿关系变化更新 `characters/relations.yaml`

### 用户主动触发才执行

只有当用户显式调用以下命令时，才执行对应更新：

| 用户意图 | 触发命令 |
|---|---|
| 用草稿更新大纲 | `/plot-edit` 或 `/plot-add` |
| 用草稿更新人物卡 | `/character-edit` |
| 用草稿更新设定 | `/setting-edit` |
| 用草稿更新时间线 | `/timeline-add` |
| 用草稿更新关系 | `/relationship-log` |
| 批量同步所有材料 | `/project-reindex`（需明确确认） |

## 冲突报告模板

```markdown
## ⚡ 草稿与结构化材料存在冲突，请决策

以下冲突已检测到，**我不会自动修改任何材料**，请你决定如何处理：

### 冲突 1：[简短标题]
- **草稿内容**：[章节 chXXX，第X段] — 具体引用草稿原文
- **当前材料**：[outline.md / characters/XXX.yaml 等] — 具体引用材料内容
- **冲突类型**：事件偏离 / 角色行为矛盾 / 设定违反 / 时间线错位（选一）
- **严重度**：轻微（不影响主线） / 显著（需要一致性补丁） / 严重（多处连锁影响）

---

### 可选操作
- 以草稿为准，更新材料 → `/plot-edit ch001 [修改内容]`
- 以材料为准，修改草稿 → 直接编辑 `chapters/chXXX.md`
- 暂时搁置，记录待办 → 告诉我，我帮你记入 TODO
```

## 处理草稿文本时的称呼保留规则

**草稿中作者使用的称呼形式就是正确的称呼形式，不得替换。**

无论是改写、润色、去 AI 感还是生成初稿，涉及草稿文本时：

- 作者在草稿中写"老王"，保留"老王"，不替换成"王某某"
- 作者在草稿中写"那个女人"，保留"那个女人"，不替换成角色的正式姓名
- 作者在草稿中写绰号、外号、代称，全部保留原样
- 角色卡中的 `aliases` 字段是参考，不是授权替换的信号

**禁止行为：**
- 因为角色卡里有正式名字，就把草稿里的非正式称呼换掉
- 因为"更清晰"就统一称呼，除非用户明确要求
- 在任何改写/润色输出中悄悄变更称呼

如果 AI 发现草稿中同一角色使用了多种称呼，只在报告中提一句（"本章赵宋同时被称为'赵宋'/'他'/'那个人'，已保留原文"），不自动统一。

## 写作流程中的使用位置

以下 skill 执行时需遵守本协议：

- `chapter-draft`：生成初稿时，若用户希望偏离大纲，标记而非强制对齐
- `pipeline-draft-polish`：打磨后输出冲突报告，不自动更新大纲
- `consistency-check`：输出冲突清单，不执行任何修复
- `chapter-review`：标记潜在大纲偏离，等待用户指令
- `project-reindex`：重建索引前，先输出冲突摘要，确认后再执行

## 例外情形

以下操作不受本协议约束（属于纯元数据，不涉及内容决策）：

- 更新章节状态（`idea → outline → draft → revise → done`）
- 更新字数统计（`word_actual`）
- 更新 `project.updated` 日期戳
