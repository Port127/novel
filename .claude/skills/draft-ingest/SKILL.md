# 任务

深度阅读用户提供的素材文档，产出一份**理解摘要（Ingestion Brief）**。
这份摘要是后续所有结构化工作（大纲、世界观、角色）的输入源，而非草稿本身。

核心原则：**先理解，再确认，最后才动手。**

## 前置检查

1. 确认 `$0`（draft_path）指向的文件存在且可读
2. 读取 `.current.yaml` 获取 `current_path`（若有当前项目）
3. 若当前项目已有 `ingestion_brief.md`，提示用户是否覆盖或追加

## 输入参数

- `$0` (draft_path): 草稿文件路径（必需）

## 执行步骤

### 1. 全文深度阅读

通读全文，**不做任何提炼**，先建立完整理解。重点关注：

- **段落间的逻辑关系**：哪些是并列设定，哪些是递进演绎
- **最后出现的信息**：草稿末尾的内容常常是作者最想表达但尚未展开的核心想法
- **隐含意图**：作者写了什么，但没有直接说出来的东西
- **矛盾与张力**：哪些设定之间存在创意张力（不是 bug，是特征）

### 2. 提取故事 DNA

**消化素材是文学解读，不是信息提取：**
- 带着"作者真正想写什么"的问题去读，不是"这段文字说了什么"
- 独特机制不是一个标签（如"时间回溯"），而是一个叙事可能性（如"每次回溯都让主角更接近真相，也更接近疯狂"）
- 角色胚胎不是一个功能位（如"导师"），而是一个人（如"一个不想当导师但被误解为导师的人"）
- 对草稿中最粗糙的部分特别留意——那往往是作者最兴奋但还没想好怎么写的核心

从阅读中提取以下维度，每条需附上**原文依据**（引用草稿中的具体段落或关键词）：

#### 2a. 真实前提（True Premise）

不是一句营销式概括，而是回答：**这个故事的核心叙事引擎是什么？**

检验标准：
- 能解释"为什么这个故事值得讲"
- 能区分这个故事和同类故事的差异
- 涵盖主角的核心困境，而非只是背景设定

#### 2b. 独特机制（Unique Mechanism）

这个故事区别于同类故事的**叙事装置**是什么？可能是：
- 一个特殊的能力规则（及其带来的叙事可能性）
- 一种特定的信息结构（如：读者知道什么，角色不知道什么）
- 一个时间/空间/身份层面的结构性悬念

#### 2c. 主角真实身份与处境

不只是"这个人是什么"，而是"这个人在故事开始时**处于什么状态**，以及**为什么处于这个状态**"。

重点区分：
- 主角"看起来是什么" vs "实际上是什么"
- 主角的外在困境 vs 内在困境
- 哪些信息是开局就揭示的，哪些是逐步揭示的

#### 2d. 世界规则提取

从草稿中识别：
- **已明确定义的规则**（原文直接写了）
- **已暗示但未展开的规则**（从描述中可推断）
- **尚未定义但故事运转所必需的规则**（规则缺口）

#### 2e. 叙事结构线索

从故事的内在逻辑中推断可能的结构，而非套模板：
- 故事自然暗示了几个阶段？
- 核心冲突的升级路径是什么？
- 主角认知的转变节点在哪？

#### 2f. 关键势力/角色胚胎

草稿中出现的角色或势力，标注其：
- 对主角的态度
- 在故事中的功能（障碍/盟友/镜像/催化剂）
- 尚未定义但需要的关键角色位

### 3. 标记理解不确定区

将以下内容显式标记为**需要用户确认**：
- 草稿中存在的模糊或矛盾之处
- 可能有多种解读的段落
- 缺失的关键信息（如主角姓名、具体时间点）
- 你认为作者可能还没想好的部分

### 4. 输出理解摘要

以对话形式输出，每个维度附带"我的理解 → 原文依据 → 确认问题"。

### 5. 等待用户确认

**在用户确认之前，不执行任何后续步骤。**

用户可能：
- 确认全部正确 → 写入 `ingestion_brief.md`
- 修正部分理解 → 更新后写入
- 补充新信息 → 合并后写入

### 6. 写入理解摘要

确认后写入 `{current_path}/ingestion_brief.md`（或 `drafts/` 旁）。
此文件是后续 pipeline 的输入源。

### 7. 更新状态

更新 `{current_path}/.novel/state.yaml`：

```yaml
ingestion:
  status: completed
  brief_file: ingestion_brief.md
  source_draft: "$0"
```

同时刷新 `project.updated` 为今天日期。

## 注意事项

- **永远不要跳过确认步骤。** 这是整个流程的核心闸门。
- 引用原文时要精确到具体句子或段落，不要泛泛地说"根据设定"。
- 草稿末尾的零散想法往往是最重要的，不要忽略。
- 理解摘要不是草稿的重新排版，而是对草稿的**分析性解读**。
- 如果草稿中存在创意性矛盾（如"普通人"但"直接获得概念核心"），先当作**设计意图**去理解，而非 bug。
